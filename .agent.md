# Agent Policy — Repository Level

Branching & commits

- Branch name conventions: `feat/<short>-<slug>`, `fix/<short>-<slug>`, `chore/<desc>`.
- Commit message templates: `feat(scope): short summary`, `fix(scope): short summary`, `chore: short summary`.

Pull requests

- PR checklist: tests pass, summary of changes, verification steps, CI green.
- Agent must open PRs; merging to `main` requires a human reviewer and explicit approval.

Permissions

- Allowed: create branches, run builds/tests, modify code and tests, add docs.
- Forbidden: commit secrets, modify GitHub secrets, merge to protected branches.

CI & gating

- All feature PRs must pass the backend `dotnet` tests and frontend `vitest` before merging.

Coding standards & tooling enforcement

- Agents must run and pass the repository pre-commit checks before opening a PR: CSharpier for C# files, ESLint + Prettier for JS/TS files, and compilation/unit tests for both projects.
- The agent should prefer non-destructive, small PRs and include a technical Markdown document for new features.

Forbidden changes

- Never change project inter-dependencies (project references) without explicit human approval. Changes to solution/project references require a maintainer review.

Pre-commit workflow (agent)

1. Run `Run CSharpier` skill when there are staged or modified C# files.
2. Run `Run ESLint` and `Run Prettier` skills for frontend files; apply fixes where available.
3. Run unit tests: `dotnet test` (Backend) and `npm run test:unit` (Client).
4. If all checks pass, create a feature branch and open a PR using the repository PR template. If any check fails, abort and add a comment describing the failure and remediation steps.
